1. One-Page Intuition

The Bank Loan Analogy

Imagine a bank with limited cash reserves (say $10,000 total). Three customers (workers) each want loans at different times. The bank manager knows each customer's maximum possible loan request in advance.

  • Customer A might need up to $7,000 total
  • Customer B might need up to $3,000 total
  • Customer C might need up to $9,000 total

A smart bank manager never gives out money that would leave the bank unable to satisfy at least one customer enough to finish their business and repay. That way, the money comes back and can serve others.

A foolish manager gives money freely → customers hold partial funds → everyone waits for everyone else → deadlock. The bank freezes.

Your project is that smart manager — the Banker's Algorithm.

What Actually Runs

When you launch the program:

  1. One Manager process (parent) starts — it owns shared memory
  2. Three Worker processes (children via fork()) start simultaneously
  3. Workers race to request resources (like customers calling the bank)
  4. Manager intercepts each request, runs Banker's Algorithm, grants or denies
  5. Workers finish, release resources, exit
  6. Manager confirms clean exit, frees memory

3. The Problem & Why It Matters

What Is Deadlock

Formal definition: A set of processes is deadlocked if every process in the set is waiting for an event that can only be caused by another process in the set — and no process can proceed.

Simple version: Two cars at a single-lane bridge from opposite sides. Neither moves. Neither will reverse. Both wait forever.

Four Necessary Conditions (Coffman, 1971)

ALL four must hold simultaneously for deadlock to occur:

#ConditionMeaning
1Mutual ExclusionResource held by only one process at a time
2Hold and WaitProcess holds a resource while waiting for another
3No PreemptionResources cannot be forcibly taken away
4Circular WaitP1 waits for P2, P2 waits for P3, P3 waits for P1

Where Deadlock Happens in Real Systems

SystemExample
Operating SystemTwo threads lock mutex A then B vs B then A
Database (MySQL/Postgres)Transaction 1 locks row A needs row B; T2 locks row B needs row A
Distributed SystemsMicroservices waiting on each other's API responses
Java/Python threadssynchronized blocks acquired in different orders
Cloud schedulersJobs holding partial GPU/CPU allocations, waiting for more

Why It's Dangerous

  • System hangs with no error message — silent failure
  • CPU usage drops to zero but processes never finish
  • Requires manual kill or reboot to recover
  • In production: bank transactions frozen, hospital systems unresponsive

7. Execution Flow — Step-by-Step

1Program starts: main() shows menu. User picks mode. Shared memory initialised via ipc_init().
2Workers forked: fork() called 3 times. Each child calls run_worker(id, shm, mode).
3Worker prepares request: Picks request vector from scenario table (or user input in manual mode).
4Worker writes to shared memory: sem_wait(mutex) → writes request[], request_owner, sets result to -1 (pending) → sem_post(mutex).
5Worker signals manager: sem_post(request_ready) — wakes the manager.
6Worker blocks: sem_wait(request_done) — sleeps until manager responds.
7Manager wakes: sem_wait(request_ready) returns. Manager locks mutex, reads request and owner.
8Check 1 — Exceeds Need? For each resource j: if request[j] > need[owner][j] → return -1.
9Check 2 — Exceeds Available? For each j: if request[j] > available[j] → return -2.
10Tentative allocation: available[j] -= request[j], allocation[owner][j] += request[j], need[owner][j] -= request[j].
11Safety check: Run is_safe_state(). If unsafe → rollback all three matrices → return 0.
12Grant: Keep allocation. Set request_result = 1. Release mutex. Print decision. Signal worker: sem_post(request_done).
13Worker wakes: Reads result, prints GRANTED/DENIED, uses resource (sleep(1)), then releases all, sets worker_done[i] = 1.
14Cleanup: Manager detects all done, exits loop. ipc_cleanup() destroys semaphores, detaches shared memory.

8. Banker's Algorithm — Intuitive + Formal

Key Concepts

TermMeaning
Safe StateThere exists at least one ordering of processes that allows all to finish without deadlock
Safe SequenceThe specific ordering, e.g., <W2, W1, W3> — each can finish using available + prior releases
Unsafe StateNo such ordering exists — deadlock is possible (not guaranteed, but possible)

Safety Algorithm (is_safe_state)

  1. Set Work = Available (copy of free resources)
  2. Set Finish[i] = false for all processes
  3. Find process i where: Finish[i] == false AND Need[i] ≤ Work
  4. If found: Work = Work + Allocation[i], Finish[i] = true, go to step 3
  5. If no such i found: stop
  6. If all Finish[i] == true → SAFE. Else → UNSAFE

Worked Safety Example

State: Available = [3, 3, 2]

WorkerAllocationNeed
W1[0,1,0][7,4,3]
W2[2,0,0][1,2,2]
W3[3,0,2][6,0,0]
Work = [3,3,2],  Finish = [F, F, F]

Round 1:
  W1 Need [7,4,3] ≤ [3,3,2]? NO
  W2 Need [1,2,2] ≤ [3,3,2]? YES → Work = [3,3,2]+[2,0,0] = [5,3,2], Finish[W2]=T

Round 2:
  W1 Need [7,4,3] ≤ [5,3,2]? NO
  W3 Need [6,0,0] ≤ [5,3,2]? NO
  (W2 already done)
  No progress → UNSAFE STATE

Safe sequence does NOT exist with this allocation.

Resource-Request Algorithm (banker_request)

  1. If Request[i] > Need[i] → error, deny (-1)
  2. If Request[i] > Available → wait, deny (-2)
  3. Tentatively: Available -= Request[i], Allocation[i] += Request[i], Need[i] -= Request[i]
  4. Run safety check → if SAFE: keep allocation, return 1
  5. If UNSAFE: rollback step 3, return 0

14. Viva Q&A — 15 Questions with Answers

Q1. What are the four necessary conditions for deadlock?

Mutual exclusion, hold and wait, no preemption, and circular wait. All four must hold simultaneously. Banker's Algorithm prevents deadlock by ensuring circular wait never develops — it refuses requests that would lead to an unsafe state.

Q2. What is the difference between deadlock avoidance and deadlock prevention?

Prevention eliminates one of the four conditions structurally (e.g., enforce resource ordering to prevent circular wait). Avoidance allows the conditions to exist but makes careful runtime decisions — Banker's Algorithm is avoidance: it checks each request at runtime and denies it if the system would enter an unsafe state.

Q3. What is a safe state? What is an unsafe state?

A safe state means there exists a safe sequence — an ordering of processes such that each can obtain its remaining resources, finish, and release them for the next. An unsafe state means no such sequence exists. Unsafe does not guarantee deadlock immediately, but it means deadlock is possible.

Q4. Why does Banker's Algorithm require max need to be declared upfront?

The safety algorithm needs to simulate whether each process can eventually finish. Without knowing the maximum it will ever request, we cannot determine if it can complete given current available resources. This is the main practical limitation of Banker's Algorithm.

Q5. What is shared memory and why do you use it?

Shared memory is a region of RAM accessible by multiple processes simultaneously — created with shmget() and attached with shmat(). It is the fastest IPC mechanism because there is no kernel copying; all processes read and write directly to the same physical memory pages. We use it so worker processes and the manager can exchange resource state without overhead.

Q6. What is a semaphore and how is it used in your project?

A semaphore is a kernel-level synchronization primitive with two operations: wait (decrement, block if zero) and post (increment, wake blocked process). We use three: mutex (value=1) for mutual exclusion on shared memory, request_ready (value=0) for worker-to-manager signaling, and request_done (value=0) for manager-to-worker signaling.

Q7. What is a race condition and how do your semaphores prevent it?

A race condition occurs when two processes access shared data concurrently and the result depends on execution order. For example, if two workers wrote their requests simultaneously, the manager would read corrupted data. The mutex semaphore ensures only one process can access shared memory at a time, preventing this.

Q8. What does fork() do and what is the difference between parent and child?

fork() creates an exact copy of the calling process. In the parent, fork() returns the child's PID. In the child, it returns 0. Both have the same memory contents initially, but execute independently. In our project, the parent becomes the manager and the children become workers — they all share the same shared memory pointer because it was attached before forking.

Q9. Why does your vulnerable mode potentially deadlock?

In vulnerable mode, all four deadlock conditions can hold: resources are exclusive (mutex), workers hold some and wait for more (hold-and-wait), the manager cannot preempt allocations (no preemption), and the hardcoded scenarios create circular dependencies (worker 1 needs what worker 2 holds and vice versa). Without Banker's check, these conditions are never broken.

Q10. Explain the Need matrix. How is it maintained?

Need[i][j] = Max[i][j] - Allocation[i][j] — the remaining resources process i still needs. It is updated in two places: when a request is granted (Need[i][j] -= request[j]), and when a worker releases all resources (Need[i][j] = Max[i][j], resetting back to full max).

Q11. What happens during rollback in banker_request?

If the safety check fails after tentative allocation, we undo all three changes: available[j] += request[j], allocation[i][j] -= request[j], need[i][j] += request[j]. This restores the system to its pre-request state so no resources are actually given. The worker is then told DENIED.

Q12. What is the time complexity of the safety algorithm?

O(P² × R) where P = number of processes, R = number of resources. The outer while loop can run at most P times (one process finishes per pass), the inner loop scans all P processes, and the need check scans R resources. For our system (P=3, R=3) this is trivially fast.

Q13. How is IPC different from regular function calls?

Regular function calls share the same process memory and stack. IPC is communication between separate processes with separate address spaces. Shared memory gives them a common region; semaphores coordinate access. Without IPC mechanisms, one process cannot read or write another's variables.

Q14. What would happen if you removed the mutex semaphore?

Multiple workers could write their requests simultaneously, corrupting request[], request_owner, and the matrices. The manager could read a mix of two workers' data. The allocation and need matrices could be inconsistent, causing wrong safety decisions and potentially granting unsafe allocations.

Q15. How would you extend this to a distributed system?

Replace shared memory with a message queue or network socket (TCP/UDP). The manager becomes a central server. Workers become clients on different machines. The challenge is distributed deadlock detection — you need a global resource-request graph across nodes. Algorithms like Chandy-Misra-Haas extend Banker's idea to distributed settings. The semaphores would be replaced by distributed locks (e.g., using Redis or ZooKeeper).
